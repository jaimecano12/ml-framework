1. INTRODUCCIÓN Y MOTIVACIÓN
________________________________________________________________________________

En los proyectos reales de Machine Learning, la calidad del dato y la integridad
del proceso de entrenamiento son factores críticos que determinan si un modelo
generaliza bien o simplemente "memoriza" el conjunto de entrenamiento. Sin
embargo, la práctica habitual en muchos equipos consiste en ejecutar directamente
un pipeline de modelado sin inspeccionar previamente el dataset, lo que deriva en
dos tipos de problemas fundamentales:

El primero es la mala calidad del dato: valores faltantes, filas duplicadas,
outliers extremos, clases desbalanceadas, columnas constantes o variables con
varianza casi nula. Cada uno de estos problemas afecta de forma distinta al
entrenamiento y puede hacer que un modelo aparentemente bueno en validación
fracase en producción.

El segundo, y más grave, es el data leakage o fuga de datos: situación en la que
información del futuro o del conjunto de test "se filtra" al conjunto de
entrenamiento. El leakage produce modelos con métricas de validación infladas que
luego no se sostienen en producción. Sus formas más comunes son: una feature
perfectamente correlacionada con el target (proxy leakage), solapamiento de filas
entre train y test, desorden temporal en series temporales, o columnas
identificadoras que memorizan el dataset.

El objetivo de este TFM es construir un framework en Python que automatice la
detección de todos estos problemas, cuantifique su impacto en el rendimiento del
modelo mediante un análisis comparativo antes/después de corregirlos, y genere
un informe HTML autocontenido que un científico de datos pueda revisar sin
necesidad de ejecutar código adicional.

________________________________________________________________________________

2. OBJETIVOS DEL PROYECTO
________________________________________________________________________________

Los objetivos concretos del framework son los siguientes:

  1. Detectar problemas de calidad del dato de forma automática y configurable.
  2. Identificar las principales formas de data leakage sin intervención manual.
  3. Cuantificar cómo los problemas detectados degradan (o inflan) el rendimiento
     del modelo mediante validación cruzada comparativa.
  4. Generar un informe HTML profesional con tablas, métricas y gráficas.
  5. Hacer todo lo anterior configurable mediante un único fichero YAML.
  6. Mantener el código testeable, modular y extensible siguiendo principios de
     ingeniería de software de producción.

3. TECNOLOGÍAS UTILIZADAS
________________________________________________________________________________

El proyecto se desarrolla íntegramente en Python 3.13 bajo un entorno Miniconda.
Las dependencias principales son:

  - pandas / numpy         — manipulación de datos y operaciones numéricas
  - scipy                  — estadística (chi-cuadrado, correlaciones)
  - scikit-learn           — modelos de ML, pipelines, validación cruzada
  - xgboost                — modelo adicional de gradient boosting
  - pyyaml                 — carga del fichero de configuración
  - jinja2                 — motor de plantillas para el informe HTML
  - matplotlib             — generación de gráficas embebidas en el informe
  - loguru                 — logging estructurado y legible
  - pytest                 — framework de tests (178 tests en total)

4. ARQUITECTURA DEL SISTEMA
________________________________________________________________________________

El framework sigue una arquitectura de módulos desacoplados donde cada fase
produce una lista de objetos CheckResult que se acumulan en un FrameworkReport.
El flujo de ejecución es siempre el mismo:

  config.yaml → load_config() → DataFrame
                                    │
                         ┌──────────┼──────────────┐
                         ▼          ▼               ▼
              quality_checks  leakage_checks  impact_analysis
                         │          │               │
                         └──────────┴───────────────┘
                                    │
                           FrameworkReport
                                    │
                          generate_report() → HTML

La clase central CheckResult encapsula el resultado de cualquier verificación
individual con los siguientes campos: check_name (identificador), passed
(booleano), severity (info / warning / error), message (texto legible),
details (diccionario con datos estructurados) y affected_columns (lista de
columnas afectadas). Esta interfaz uniforme permite que el módulo de informes
procese cualquier resultado sin conocer qué check lo generó.

El FrameworkReport agrega todas las listas de resultados y expone métodos de
conveniencia: all_results(), failed_checks() y summary(), que devuelven
estadísticas consolidadas de toda la ejecución.

La entrada de usuario es siempre un fichero YAML cuya estructura está validada
por src/config.py antes de ejecutar ningún check. Errores de configuración
(threshold fuera de rango, métrica desconocida, etc.) se reportan todos juntos
al arranque, no uno a uno durante la ejecución.

5. FASE 1 — CONFIGURACIÓN INICIAL DEL PROYECTO
________________________________________________________________________________

La primera fase establece el andamiaje completo del proyecto: estructura de
directorios, dependencias, módulos esqueleto y tests de humo.

La estructura de directorios separa claramente datos (data/), configuración
(configs/), código fuente (src/), tests (tests/), informes generados (reports/)
y scripts de experimentación (scripts/).

El fichero src/utils.py implementa tres componentes reutilizables por el resto
de módulos:

  - setup_logger(): configura loguru con formato coloreado y nivel configurable,
    con soporte opcional para un fichero de log rotativo.

  - load_dataset(): carga un DataFrame desde CSV, Parquet o Excel, con
    detección automática de formato por extensión y propagación de errores
    comprensibles (FileNotFoundError para rutas inexistentes, ValueError para
    formatos no soportados).

  - CheckResult y FrameworkReport: dos dataclasses tipadas que constituyen la
    columna vertebral del framework. CheckResult valida su campo severity en
    __post_init__ para evitar valores inválidos en tiempo de ejecución.

El punto de entrada principal, main.py, expone una CLI con argparse que acepta
--config, --dataset, --output-dir y --log-level, permitiendo ejecutar el
framework desde línea de comandos sin modificar ningún fichero de código.

Los 16 tests de esta fase verifican que el logger no lanza excepciones, que
load_dataset maneja correctamente CSV y Parquet, que CheckResult rechaza
severidades inválidas, y que FrameworkReport agrega y contabiliza resultados
correctamente.

________________________________________________________________________________

6. FASE 2 — SISTEMA DE CONFIGURACIÓN YAML
________________________________________________________________________________

El fichero configs/config.yaml centraliza todos los parámetros del framework:
ruta del dataset, columna objetivo, umbrales de cada check, modelos a evaluar,
métricas de rendimiento y configuración del informe.

La función load_config() en src/config.py implementa un patrón de merge
profundo (deep merge) entre un diccionario de valores por defecto (_DEFAULTS)
y el YAML del usuario. Esto significa que un fichero de configuración mínimo
(que sólo especifique dataset.path y dataset.target_column) funciona
correctamente porque todos los demás valores se toman de _DEFAULTS. El usuario
sólo necesita especificar lo que quiere cambiar.

La validación es exhaustiva y colecta todos los errores antes de lanzarlos
juntos, evitando el ciclo frustrante de corregir un error, ejecutar de nuevo,
descubrir el siguiente. Se validan: campos obligatorios presentes, niveles de
log válidos, métodos de detección de outliers, thresholds en rango [0,1],
modelos conocidos, métricas conocidas, cv_folds >= 2, test_size en (0,1), y
formato de informe soportado.

La helper get_section(config, section) permite extraer bloques específicos del
config para pasarlos a cada módulo de checks, manteniendo el acoplamiento bajo.

Los 26 tests de esta fase cubren la carga exitosa, la aplicación de defaults,
la detección de errores múltiples simultáneos, y la función de deep merge
(incluyendo el caso edge de que las listas se reemplazan, no se fusionan).

7. FASE 3 — MÓDULO DE CALIDAD DE DATOS
________________________________________________________________________________

El módulo src/quality_checks.py implementa seis verificaciones de calidad, cada
una devolviendo un CheckResult con severidad escalable según la gravedad del
hallazgo.

CHECK 1 — VALORES FALTANTES
Calcula la tasa de NaN por columna y señala las que superen el umbral configurado
(por defecto 5%). La severidad sube a "error" cuando una columna supera el 50%
de valores faltantes, pues en ese punto la columna aporta más ruido que señal.

CHECK 2 — FILAS DUPLICADAS
Detecta filas exactamente duplicadas y calcula su tasa sobre el total. La
severidad escala a "error" cuando la tasa supera el 10%, indicando un problema
sistemático en la recopilación de datos.

CHECK 3 — OUTLIERS
Soporta dos métodos configurables. El método IQR (por defecto) define como
outlier cualquier valor fuera del rango [Q1 - k·IQR, Q3 + k·IQR] siendo k el
threshold configurado (por defecto 3.0). El método z-score marca como outlier
cualquier valor con |z| > threshold. El check omite columnas con menos de 4
valores (IQR no es estadísticamente representativo) y columnas constantes
(donde IQR = 0, lo que provocaría una división por cero).

CHECK 4 — DESBALANCE DE CLASES
Calcula la distribución de frecuencias del target y señala si la clase
minoritaria está por debajo del umbral (por defecto 10%). Funciona tanto para
clasificación binaria como multiclase. La severidad sube a "error" cuando la
clase minoritaria representa menos del 5%, situación en la que la mayoría de
métricas de rendimiento resultan engañosas.

CHECK 5 — FEATURES CONSTANTES
Identifica columnas donde todos los valores no nulos son iguales. Estas
columnas no aportan información al modelo pero sí consumen recursos y pueden
causar problemas numéricos en algunos algoritmos.

CHECK 6 — BAJA VARIANZA
Detecta columnas numéricas con coeficiente de variación (CV = std/|media|) por
debajo del umbral configurado. El CV es una medida relativa de dispersión que
no depende de la escala de la variable, lo que lo hace más adecuado que la
varianza absoluta para comparar columnas con diferentes unidades.

DECISIÓN DE DISEÑO RELEVANTE: en la implementación inicial se utilizó varianza
normalizada min-max para este check, pero se descubrió durante los tests que
la normalización min-max siempre proyecta los valores al rango [0,1],
asignando varianza similar a columnas casi-constantes y a columnas con
distribución genuinamente dispersa. Se corrigió sustituyendo por el coeficiente
de variación, que sí captura la "cercanía a la constante" de forma
escala-independiente.

________________________________________________________________________________

8. FASE 4 — DETECCIÓN DE DATA LEAKAGE (FASE CRÍTICA)
________________________________________________________________________________

Esta fase es la más importante del proyecto desde el punto de vista académico,
ya que la detección automática de data leakage es un problema poco abordado en
la literatura práctica de ML.

CHECK 1 — TARGET LEAKAGE
Detecta features que están sospechosamente correlacionadas con la variable
objetivo. Para features numéricas se utiliza la correlación de Pearson en
valor absoluto; para features categóricas se emplea la V de Cramér, una
medida de asociación basada en la estadística chi-cuadrado normalizada.
El check excluye automáticamente la columna target de la búsqueda. La
severidad es siempre "error" porque cualquier feature con correlación >= 0.95
casi con certeza está derivada del target o lo replica.

La función auxiliar _feature_target_association() maneja los cuatro casos
posibles (numérico-numérico, numérico-categórico, categórico-numérico,
categórico-categórico) con el estadístico apropiado en cada caso, devolviendo
0.0 cuando hay datos insuficientes o la computación falla.

CHECK 2 — SOLAPAMIENTO TRAIN/TEST
Simula un split aleatorio con los parámetros configurados (test_size, random_state)
y comprueba si alguna fila aparece en ambas particiones. Esto sólo es posible
cuando existen filas duplicadas. El check es útil para detectar situaciones
donde un dataset fue ensamblado concatenando train y test de una competición
sin eliminar duplicados.

CHECK 3 — LEAKAGE TEMPORAL
Cuando se configura una columna de fecha (date_column), verifica que el dataset
esté ordenado cronológicamente. Un dataset desordenado temporalmente hace que
un split secuencial mezcle observaciones del futuro con el pasado, permitiendo
al modelo "ver" información que en un despliegue real no estaría disponible.
El check también detecta fechas no parseables que podrían indicar corrupción
de datos.

CHECK 4 — COLUMNAS IDENTIFICADORAS
Detecta columnas de tipo string o entero cuya proporción de valores únicos
supera el umbral configurado (por defecto 95%). Columnas como IDs de cliente,
UUIDs o números de transacción tienen cardinalidad máxima y permiten al modelo
memorizar el conjunto de entrenamiento en lugar de aprender patrones generales.

DECISIÓN DE DISEÑO RELEVANTE: en la versión inicial, este check marcaba
también columnas numéricas continuas (como precios o ingresos) cuya
cardinalidad es naturalmente alta. Se corrigió excluyendo columnas de tipo
float, ya que la alta cardinalidad en variables continuas es un comportamiento
esperado y no indica un problema de leakage.

9. FASE 5 — ANÁLISIS DE IMPACTO
________________________________________________________________________________

El módulo src/impact_analysis.py cuantifica el efecto real de los problemas
detectados sobre el rendimiento del modelo, respondiendo a la pregunta: "¿En
cuánto inflaban estas features problemáticas la métrica de validación?"

El proceso es el siguiente para cada modelo configurado:

  1. BASELINE: se entrena el modelo sobre el dataset completo con validación
     cruzada estratificada de k pliegues. Este es el rendimiento "contaminado"
     que incluye todas las features problemáticas.

  2. CLEANED: se eliminan del dataset las columnas marcadas por target_leakage,
     id_column_leakage y constant_features; opcionalmente se eliminan filas
     duplicadas. Se vuelve a entrenar con validación cruzada.

  3. DELTA: se calcula la diferencia cleaned - baseline. Un delta negativo
     revela que el modelo se apoyaba en features problemáticas para inflar
     su métrica. Un delta de -0.05 o menor activa el flag passed=False.

El preprocesamiento es automático: pipeline de SimpleImputer (media) →
StandardScaler → modelo, todo encapsulado en un sklearn Pipeline para evitar
data leakage en la propia validación cruzada. Las features categóricas se
codifican con factorize() antes del pipeline. El scorer se adapta
automáticamente a clasificación binaria o multiclase (roc_auc se convierte en
roc_auc_ovr_weighted para multiclase).

La función _extract_problem_columns() extrae del FrameworkReport las columnas
afectadas por checks fallidos del tipo target_leakage, id_column_leakage y
constant_features, sin ninguna lógica adicional: simplemente lee los
affected_columns de los resultados previos. Esto ilustra la arquitectura
orientada a datos del framework: cada fase consume los resultados de la
anterior.

________________________________________________________________________________

10. FASE 6 — GENERACIÓN DE INFORMES HTML
________________________________________________________________________________

El módulo src/reporting.py transforma el FrameworkReport en un fichero HTML
autocontenido (sin dependencias externas de CSS ni JavaScript) que cualquier
persona puede abrir en un navegador sin instalar nada.

La plantilla Jinja2 (src/templates/report.html.j2) está estructurada en
secciones claras: cabecera con metadatos del dataset, tarjetas de resumen
(total de checks, pasados, fallidos, errores, warnings), tabla de quality checks,
tabla de leakage checks, tabla de impact analysis con coloración condicional del
delta (verde para mejora, rojo para degradación), y sección de visualizaciones.

El módulo genera tres gráficas automáticas con matplotlib, codificadas como
imágenes PNG en base64 e incrustadas directamente en el HTML:

  - Diagrama de barras horizontales: checks pasados vs. fallidos por fase
    (Quality, Leakage, Impact).
  - Diagrama de tarta: distribución de severidades (info / warning / error)
    sobre el total de checks ejecutados.
  - Diagrama de barras agrupadas: comparación baseline vs. cleaned para
    cada modelo en el análisis de impacto.

El nombre del fichero de salida incluye el nombre del dataset y el timestamp de
ejecución, permitiendo conservar histórico de ejecuciones sin sobreescribir.

11. FASE 7 — EXPERIMENTOS Y EVALUACIÓN
________________________________________________________________________________

La fase final valida el framework sobre datos reales y sintéticos diseñados para
ejercitar todos los checks implementados.

GENERACIÓN DE DATOS SINTÉTICOS (scripts/generate_data.py)
Se generaron tres datasets con características controladas:

  - clean_dataset.csv (500 filas × 6 columnas): sin ningún problema de calidad
    ni leakage. Sirve como control para verificar que el framework no genera
    falsos positivos.

  - dirty_dataset.csv (~5300 filas × 8 columnas): incluye 25% de valores
    faltantes en dos columnas, 30 filas duplicadas, 10 outliers extremos en f3,
    desbalance de clases (5% clase minoritaria), una columna constante y una
    columna de baja varianza.

  - leaky_dataset.csv (~5320 filas × 12 columnas): incluye todo lo anterior
    más una feature "target_proxy" idéntica al target, una "noisy_proxy"
    con correlación ~0.98, una columna "user_id" como identificador único, y
    una columna "event_date" con fechas desordenadas.

RESULTADOS DEL EXPERIMENTO
Los resultados obtenidos confirman que el framework detecta correctamente todos
los problemas introducidos:

  Dataset            | Checks totales | Pasados | Fallidos | Δ accuracy (LR)
  -------------------|----------------|---------|----------|----------------
  clean_dataset      |      11        |   11    |    0     |    +0.000
  dirty_dataset      |      11        |    6    |    5     |    +0.000
  leaky_dataset      |      11        |    4    |    7     |    −0.048

El resultado más relevante es el de leaky_dataset: el modelo de regresión
logística alcanza accuracy=1.000 cuando incluye las features de leakage, que
cae a 0.952 al eliminarlas. Esta diferencia de 4.8 puntos porcentuales
cuantifica exactamente el efecto del leakage sobre la métrica de validación.

12. ERRORES ENCONTRADOS Y CÓMO SE CORRIGIERON
________________________________________________________________________________

Durante el desarrollo se encontraron varios errores técnicos y de diseño que
merecen documentarse por su valor pedagógico.

ERROR 1 — CONFLICTO DE ENTORNO PYTHON
Síntoma: el comando pip instalaba paquetes en Python 3.12 pero el intérprete
activo era Python 3.13 (Miniconda), por lo que pytest y loguru no se encontraban.
Solución: uso consistente de "python -m pip" y "python -m pytest" en lugar de
los comandos directos "pip" y "pytest", que apuntaban a otro entorno.

ERROR 2 — OUTLIERS NO DETECTADOS POR IQR EN DISTRIBUCIÓN DEGENERADA
Síntoma: el test test_iqr_detects_extreme_values fallaba porque el fixture de
prueba usaba 95 valores idénticos (10) y 5 outliers extremos. Con 95% de los
valores iguales, Q1 = Q3 = 10, IQR = 0, y el método devolvía cero outliers.
Solución: se rediseñó el fixture para usar una distribución normal como base
(rng.normal(loc=10, scale=2, size=95)), garantizando que IQR > 0.

ERROR 3 — VARIANZA NORMALIZADA NO CAPTURA COLUMNAS CASI-CONSTANTES
Síntoma: el test test_detects_near_constant_column fallaba porque la normalización
min-max proyecta el rango completo de la columna a [0,1], resultando en varianza
similar para columnas casi-constantes y columnas con dispersión real.
Solución: se sustituyó la varianza normalizada por el coeficiente de variación
(CV = std / |media|), que es una medida relativa de dispersión escala-independiente.
Una columna con valores 1.0 ± 0.0001 tiene CV ≈ 0.0001 << 0.01 (umbral), mientras
que una columna normal con media=10 y std=3 tiene CV ≈ 0.3, bien por encima.

ERROR 4 — ID COLUMN LEAKAGE MARCANDO FEATURES CONTINUAS
Síntoma: el check id_column_leakage marcaba columnas numéricas continuas
(ingresos, precios) como potenciales columnas identificadoras porque su
cardinalidad es naturalmente alta (cada float es probablemente único).
Solución: se añadió una exclusión explícita para columnas de tipo float, ya que
sólo columnas de tipo string o integer pueden ser identificadores de fila.

ERROR 5 — SEVERIDAD INCORRECTA EN TEST DE DUPLICADOS
Síntoma: el test test_severity_warning_for_low_dup_rate esperaba severity="warning"
pero el fixture df_dupes tenía 3/8 = 37.5% de duplicados (> 10%), lo que
activaba severity="error".
Solución: se creó un fixture inline con sólo 1/21 = 4.7% de duplicados,
garantizando que la tasa está por debajo del umbral del 10% para "error".

ERROR 6 — DEPRECACIÓN DE datetime.utcnow() EN PYTHON 3.13
Síntoma: los tests de FrameworkReport mostraban DeprecationWarning porque
datetime.utcnow() está marcado para eliminación en Python 3.12+.
Solución: se sustituyó por datetime.now(timezone.utc), que devuelve un objeto
datetime timezone-aware equivalente pero sin deprecación.

13. VALIDACIÓN CON DATASETS REALES
________________________________________________________________________________

Para validar que el framework funciona con datos reales y no sólo con datos
sintéticos, se ejecutó el pipeline sobre el dataset Titanic (Kaggle) y el
dataset Pima Indians Diabetes (UCI/Kaggle), descargados automáticamente via
la API de OpenML de scikit-learn.

El dataset Titanic (1309 filas × 9 columnas útiles) activó los checks de
valores faltantes (Age: 20%, Embarked: 0.2%), desbalance de clases (38%
supervivientes) y columnas identificadoras (Name, Ticket con alta cardinalidad).

El dataset Pima Indians Diabetes (768 filas × 9 columnas) activó los checks de
outliers (valores extremos en Insulin y SkinThickness) y desbalance de clases
(35% diabéticos).

En ambos casos el framework generó su informe HTML en segundos, demostrando que
el tiempo de ejecución es perfectamente razonable para su uso en entornos reales.

________________________________________________________________________________

14. CONCLUSIONES
________________________________________________________________________________

El framework desarrollado cumple todos los objetivos planteados:

  1. Automatiza la detección de 10 tipos diferentes de problemas de calidad y
     leakage sobre cualquier dataset en formato CSV, Parquet o Excel.
  2. Cuantifica el impacto de estos problemas sobre la métrica de validación
     mediante análisis baseline vs. cleaned con validación cruzada.
  3. Genera un informe HTML profesional y autocontenido con gráficas embebidas.
  4. Es completamente configurable via YAML sin modificar código.
  5. Está respaldado por 178 tests automáticos que cubren todos los módulos y
     casos límite identificados durante el desarrollo.

El resultado experimental más significativo es la cuantificación del leakage:
el framework es capaz de demostrar numéricamente que un modelo con accuracy=1.000
en validación cruzada obtiene sólo 0.952 cuando se eliminan las features
problemáticas. Este tipo de evidencia es exactamente lo que un equipo de ML
necesita para justificar la inversión en calidad del dato antes del modelado.

Las posibles extensiones futuras del framework incluyen: soporte para regresión
(no sólo clasificación), detección de concept drift entre particiones temporales,
sugerencias automáticas de corrección (no sólo detección), e integración con
plataformas MLOps como MLflow o DVC para registrar los resultados de cada
ejecución como artefactos del experimento.
